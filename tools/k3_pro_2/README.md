# K3 BLE PRO 2 reverse-engineering tools

Scripts used to reverse-engineer the btScyChannel ("new security") BLE
protocol for the Tuya K3 BLE PRO 2 (`productId=ba2qk177`, category
`jtmspro`). They are kept standalone (no Home Assistant dependency) so
you can iterate on the protocol without reinstalling the integration.

The core finding: these locks use `sec_flag=14` for `DEVICE_INFO` and
`sec_flag=15` for everything after `PAIR`, with keys derived from
`MD5(local_key + sec_key [+ srand])` instead of the legacy
`MD5(login_key[:6] [+ srand])`. The DP71 unlock payload format is
`[member=0xFFFF][ver=0x0001][check_code:8][action:1][ts:4][0x0001]`
(different from the report echo direction).

## Files

- **`fetch_device_info.py`** — logs into the Tuya mobile API and prints
  export statements for `LOCAL_KEY`, `SEC_KEY`, `DEVICE_UUID`,
  `DEVICE_ID`, and the current `CHECK_CODE` (parsed from cloud DP71).
  All three rotate when the lock is re-paired, so re-run after any
  factory reset / re-pair.
- **`ble_unlock.py`** — connects to the lock, performs the full btsc
  handshake, and sends a DP71 lock/unlock. Useful for sanity-checking
  the protocol outside the integration.
- **`decode_sniff.py`** — decrypts a `btsnoop_hci.log` capture using the
  same key derivation. Unlike the generic `../decode_btsnoop.py` it
  supports sec_flags 14/15.

## Typical workflow

```sh
# 1. Fetch current keys + check code from cloud
export TUYA_USERNAME='you@example.com'
export TUYA_PASSWORD='...'
eval "$(python3 fetch_device_info.py)"

# 2. On macOS, look up the Core Bluetooth peripheral UUID once and export it.
#    On Linux/Android use the MAC directly (also printed above as a comment).
export LOCK_ADDR='XX:XX:XX:XX:XX:XX'   # or CoreBluetooth UUID on macOS

# 3. Trigger the motor
python3 ble_unlock.py          # unlock
python3 ble_unlock.py --lock   # lock
```

## Capturing traffic for analysis

On Android: Developer Options → enable **Bluetooth HCI snoop log**, do
the unlock in the Tuya app, then:

```sh
adb pull /data/misc/bluetooth/logs/btsnoop_hci.log
python3 decode_sniff.py btsnoop_hci.log \
  --local-key "$LOCAL_KEY" --sec-key "$SEC_KEY" \
  --mac DC:23:51:D1:85:77
```

The decoded output will show the DP71 write the app sent and the DP47
(motor_state) report that follows — which is how the unlock payload
format was confirmed.

## Notes

- The check code (DP71's 8-digit ASCII field) is not the PIN a user
  types; it's an internal per-device code generated by the Tuya
  backend. `fetch_device_info.py` extracts the current value from the
  cloud-mirrored DP71 record.
- On macOS, `bleak` uses the Core Bluetooth peripheral UUID (stable per
  Mac) instead of the hardware MAC for `BleakClient(address)`. Obtain
  it once with a scan, then reuse it in `LOCK_ADDR`.
- `fetch_device_info.py` speaks the `a1.tuyaXX.com` mobile API (same
  endpoint the official Android app uses). The `CLIENT_ID`/`APP_SECRET`
  constants are impersonated — use responsibly against your own
  account.
